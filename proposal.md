## Short Summary

Briefly describe the purpose of the proposal and the software project(s) it involves.

LLMs have changed the ways in which researchers learn about and use open source infrastructure. Instead of learning about a tool by Googling and following instructions, researchers increasingly rely on the latent or harness-driven knowledge of a Large Language Model (LLM) to both learn and generate code for their work. Traditionally, documentation engines provide the _interface for learning_ that a researcher uses. As the scientific community increasingly relies on LLMs, it is crucial that documentation engines also provide a _structured, machine-readable foundation_ that LLMs can leverage to accelerate learning and discovery.

MyST is a documentation engine developed by Project Jupyter. It is the engine behind [Jupyter Book](LINK), which powers over 18,000 publicly available books, courses, and knowledge bases across the sciences [TODO: Pulled this from the paper, we should link something like a github search]. MyST has an excellent foundation for the age of LLMs, because its content is _structured data by design_: the structure of every MyST document is backed by a formal specification, and the content of each page published with MyST is available as a machine-readable JSON file to encourage re-use, interoperability, and discoverability. Where most documentation is unstructured text that an LLM must scrape and guess at, MyST content can be read and re-used programmatically.

MyST already has demonstrated adoption across the scientific community, including life-science projects like the QIIME 2 Framework and NeuroLibre [TODO: LINK THESE], and its modern architecture gives us a strong, flexible base to build on. There is substantial opportunity to improve the MyST ecosystem in ways that increase the ease and power of LLM-assisted workflows. This proposal aims to make key foundational improvements in MyST's functionality in order to make it the premier documentation engine for computational narratives in the age of LLMs.

We break the work into several key areas of development:

- Improving the power and flexibility of plugin APIs. Plugins are a key part of MyST's technical strategy, as they allow authors to extend functionality in privileged ways without requiring core changes to MyST. By significantly expanding the ways in which software can _extend MyST_, plugins generated by LLMs will be more reliable, more efficient, and more powerful.
- Improving interactive widget support. LLMs have vastly reduced the barrier to building interactive data dashboards and visualizations using building block libraries like ipywidgets, panel, HoloViz, and plotly. By building functionality into MyST to more natively leverage the APIs of these tools, MyST can be a platform for rapidly creating and sharing data-driven interactive dashboards with LLMs.
- Scaling document execution to the cloud and GPUs. MyST treats execution as part of its core model: the document itself describes the computation behind a page. As more researchers bring AI into their own computation, that execution will increasingly need to scale beyond a local laptop and onto the cloud. By building native support for remote kernel execution with GPU integration via JupyterHub, computation in a MyST document can run on the cloud infrastructure that researchers already use. This gives the compute- and data-intensive workflows common in the life sciences access to far more computational power than a local machine.
- LLM-native data structures. LLMs are able to parse certain types of data more effectively in order to learn from them. For multi-page documentation sites (like those often published with MyST), a standard, programmatic way to discover all pages in a site that follow standards used by LLM workflows will allow for more efficient discovery and learning for LLM workflows. This will make it much easier for LLMs to incorporate the latest documentation for a site published with MyST in its context.
